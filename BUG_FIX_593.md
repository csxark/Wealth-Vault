# Bug Fix #593: Access tokens from tokenService are incompatible with auth middleware

## Issue
`tokenService.js` signs access tokens with payload fields like:

- `userId`
- `sessionId`
- `type: 'access'`

But the `protect` middleware in `backend/middleware/auth.js` validated tokens using `decoded.id` only.

This caused valid access tokens generated by `generateAccessToken` / `createDeviceSession` to fail authentication with `Invalid token format` because `decoded.id` was undefined.

## Root Cause
Token payload field mismatch:

- producer (`tokenService`) uses `userId`
- consumer (`protect`) expected `id`

## Fix Applied
Updated `protect` middleware in `backend/middleware/auth.js` to support both formats for compatibility:

- `const tokenUserId = decoded?.id || decoded?.userId;`

Then used `tokenUserId` for:

- token-format validation
- user lookup in database

## Expected Behavior After Fix
Protected routes now accept access tokens generated by `tokenService` while preserving compatibility with older/legacy token payloads.

## Affected Paths
- `backend/services/tokenService.js` (producer; unchanged)
- `backend/middleware/auth.js` (consumer; fixed)
